Building an HTML5 Canvas Game

This game took me longer to make than I care to admit. Below I elaborate on the things I learned.

Prerequisites

It's a good idea to familiarize yourself with some basic concepts regarding JS and the HTML5 canvas before beginning.

Setting the foundation

If you're building a web game these days, it should be responsive. Mobile usage is only going to increase as more users get access to powerful smartphones that make searches easier than ever.

Unfortunately, when I first built this game, I wasn't aware of the mobile-first approach, and it made development a hell of a whole harder than it needed to be. Designing a game that fit on viewports with a min-width: 768px is very simple; you don't have to worry about whether the actual game would fit in a landscape or portrait viewport.

It's when you're trying to the main content fit on a screen whose limiting dimension is only 320 pixels wide (I'm not sure why anyone would design for less these days). Anyway, once I finished the desktop version, the code didn't transfer so gracefully over to smaller viewports. Worse yet, I was using an HTML5 canvas element, which doesn't have a whole lot written about when it comes to responive design.

I wrote a long article going into as much detail as you'll need to make a completely responsive HTML5 canvas using PURE JavaScript.

The code

My code looks horrible. It's ~1000 lines of unoptomized, inelegant JavaScript. It's a horrible, recursive, global variable mess. Trying to re-write it now wouldn't be worth the effort because in the future I plan on making another game anyhow using what I've learned from this experience.

Attempting to decipher every huge chunk of code that makes my game wouldn't be as fruitful as describing what I accomplished:

  • An original method for making a responsive canvas. I went through a metric **** ton of different ways to try to accomplish my goal, but I learned why things worked the way they did.
  • Quick responsiveness for user selection on tablet and mobile devices. The default way mobile browsers take into account user selection is through the click event. My game responds to first contact with the screen, whether the user is on a laptop or desktop with touch-screen functionality, a tablet, or a mobile device, for maximum responsiveness.
  • Figuring out a simple algorithm for detecting adjacency.
  • Preventing easy-to-memorize patterns. This is far from elegant. The furthest. But it works.
  • Basic design considerations. This includes a point system, checking to see if the user selected the correct block, and a countdown before the game starts.

Conclusions

Learning through implementation is key, even if the implementation is completely horrid, you can never truly understand why until you get your hands dirty and try. And I tried.
